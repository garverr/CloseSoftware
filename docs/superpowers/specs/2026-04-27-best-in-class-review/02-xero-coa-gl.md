## Pillar Verdict: Xero Import (Pillar 1)

🔴 **Blocked** — The GL sync pipeline has three critical data-loss risks (single-page fetch drops journals, contact is never captured, voided entries are not flagged) and the COA type/class assignment is driven by a sign heuristic rather than Xero's authoritative account registry, making the ledger an unreliable source of truth for flux and close.

---

## Top 3 Fixes

1. **Paginate incremental journal sync** (`XeroLedgerServices.cs:560–586`): Add a `while (page.Count > 0)` loop inside `SyncTenantAsync` so all new journals since the last cursor position are fetched, not just the first 100.

2. **Capture Xero account `Type`/`Class` from the COA** (`XeroBackfillServices.cs:790–791` + `FinancialServices.cs:2135`): Add a COA sync step to the ledger pipeline using `/Accounts?includeArchived=true`; persist Xero's `Type`, `Class`, and `Status` instead of deriving them from net-amount sign.

3. **Capture contact/vendor and void flag on journal lines** (`XeroLedgerServices.cs:847–856`): Extend `XeroJournalLine` with `ContactId`, `ContactName`, and add `IsVoided` to `XeroJournal`; without contact, vendor-level flux drill-down in the AI investigation path is impossible.
