# Agent 01: Xero-Auth

**Categories:** 1, 2

## Findings

### Category 1: OAuth lifecycle, token refresh, revocation

| Severity | Finding | Evidence | Best-in-class gap | Recommendation |
| --- | --- | --- | --- | --- |
| **Blocker** | No token revocation endpoint and no disconnect flow. There is no `DELETE /api/xero/connections/{id}` route and no call to `https://identity.xero.com/connect/revocation`. A user can "disconnect" only by re-connecting; stale tokens live forever in the DB and in Xero's authorization server. | `Program.cs`: all `MapGet`/`MapPost` Xero routes listed, zero `MapDelete` for connections. No `revoke`/`disconnect` string anywhere in services. | Fathom, Closecore all call Xero's revocation endpoint and remove the DB record on user-initiated disconnect. Xero's own docs require revocation on uninstall. | Add `DELETE /api/xero/connections/{id}` that calls `POST https://identity.xero.com/connect/revocation` with `token_type_hint=refresh_token`, then deletes the `XeroTenantConnection` and `XeroConnection` rows. |
| **Blocker** | Concurrent token refresh race condition — no locking. `EnsureValidTokenAsync` in `FinancialServices.cs`, `XeroLedgerServices.cs`, and `XeroBackfillServices.cs` all do a read-then-refresh with no mutex. If the background ledger worker and a manual sync both fire for the same tenant at the same time, they each consume the current refresh token and one will fail because Xero invalidates the previous refresh token on rotation. | `FinancialServices.cs:2887-2923`; `XeroLedgerServices.cs:609-634`; `XeroBackfillServices.cs:1126-1170`. `XeroApiRequestScheduler` has a `SemaphoreSlim` but only for API call rate, not for token refresh. | Production integrations (Fathom, Numeric) use a per-tenant `SemaphoreSlim` or `IMemoryCache`-keyed lock around the refresh window, double-checking the expiry after acquiring the lock. | Introduce a `ConcurrentDictionary<string, SemaphoreSlim>` keyed by `TenantId` in a shared singleton service. All three `EnsureValidTokenAsync` implementations must acquire that lock, re-read `TokenExpiresAt` after acquisition, and only refresh if still expired. |
| **Major** | `Unprotect` has a silent plaintext fallback. When `CryptographicException` fires (wrong key-ring), all three service implementations silently return the raw ciphertext if it starts with `"ey"`. JWT access tokens always start with `"ey"`, so a mis-keyed token is returned as plaintext and then sent to Xero — the call succeeds on a stolen/leaked token even if the key-ring is wrong, and the encrypted refresh token is stored but never re-encrypted. | `FinancialServices.cs:2963`; `XeroLedgerServices.cs:801`; `XeroBackfillServices.cs:1222`. Same pattern in all three files. | Best-in-class: throw on decryption failure, mark connection `NeedsReconnect`, never return cleartext. | Remove the `"ey"` fallback. In the `catch (CryptographicException)` block, set `ConnectionStatus = "NeedsReconnect"` and `throw`, so the decryption failure surfaces rather than passing a possibly-cleartext credential to Xero. |
| **Major** | Expired `XeroOAuthSession` rows are never deleted. The table grows forever; there is no scheduled purge in `RealDataCleanupService` or anywhere else. A very old expired-but-unconsumed session with `State` collision could theoretically match a replay. | `Data/AppDbContext.cs:40,77`; `Data/RealDataCleanupService.cs` — no reference to `XeroOAuthSession`. `CompleteCallbackAsync` marks `CodeConsumedAt` but does not delete. | Best-in-class: delete consumed/expired sessions immediately or via a nightly job. | In `CompleteCallbackAsync` (`FinancialServices.cs:1749`), delete the session row after marking `CodeConsumedAt`. Add a weekly `DELETE FROM XeroOAuthSessions WHERE ExpiresAt < NOW() - 7 days` in the cleanup worker. |
| **Major** | `IsTransient` in `XeroApiRequestScheduler` does not include HTTP 401. A revoked or expired token that returns 401 from Xero triggers a generic exception rather than a refresh-then-retry cycle. The ledger sync marks the tenant `"Failed"` and exits; the user must manually reconnect. | `XeroBackfillServices.cs:116-121` — `IsTransient` includes 408, 500, 502, 503, 504, but not 401 or 403. Scheduler `GetStringAsync` does not call `EnsureValidTokenAsync`. | Fathom/Numeric: on 401, attempt one token refresh and replay the request before failing. | Detect `HttpStatusCode.Unauthorized` in `GetStringAsync`; surface it as a typed exception so the calling tenant sync can call `EnsureValidTokenAsync`, refresh, rebuild the `Authorization` header, and retry once. |
| **Moderate** | Scope set is overly broad. `appsettings.json` requests 11 scopes including `accounting.attachments.read`, `accounting.contacts.read`, and `openid profile email`, none of which are used by any current data-fetch path. | `appsettings.json:20`; `appsettings.Development.json:24`. No code calls attachments or contacts endpoints. | Xero's best-practice (and Fathom's published scope list) is minimum required scopes. Broader scopes increase blast radius on token theft. | Reduce requested scopes to `offline_access accounting.transactions.read accounting.journals.read accounting.reports.read accounting.settings.read`. Remove `openid profile email accounting.attachments.read accounting.contacts.read accounting.transactions` (write variant is not needed). |
| **Minor** | `ClientId` is committed in plaintext in both `appsettings.json` and `appsettings.Development.json`. Although not a secret for PKCE-only public clients, it is an indicator that the key-management story is underdeveloped for production. | `appsettings.json:18`; `appsettings.Development.json:18`. | Production deployments should inject via environment variables or Azure Key Vault. | Move `Xero:ClientId` to an environment variable (`XERO__CLIENTID`) and remove from committed config files. |

### Category 2: Multi-tenant connection mgmt & V2 import

| Severity | Finding | Evidence | Best-in-class gap | Recommendation |
| --- | --- | --- | --- | --- |
| **Blocker** | V2 import path in `XeroIntegrationService.ImportFinanceAppV2TokensAsync` is a permanent stub that always returns a failure message. The real import is only handled in `XeroTenantLedgerService.ImportFinanceAppV2TokensAsync`. The `POST /api/xero/import-v2-tokens` route calls `ledger.ImportFinanceAppV2TokensAsync`, but `GET /api/xero/status` reports `tokenImport.supported = true` using the stub's metadata while the ledger service does the real work — this is a split-brain surface that will confuse operators. | `FinancialServices.cs:2091-2101` (stub, always returns 0); `Program.cs:1700` calls `ledger.ImportFinanceAppV2TokensAsync`; `FinancialServices.cs:1592-1598` reports `supported = true`. | N/A — this is an internal consistency issue unique to the dual-service design. | Delete or document `XeroIntegrationService.ImportFinanceAppV2TokensAsync`; delegate status reporting to `XeroTenantLedgerService.GetStatus()` so there is one source of truth. |
| **Major** | DataProtection app-name mismatch risk is silently swallowed. The V2 import reads rows from `financeapp.db`, calls `Unprotect`, and if it fails returns `"Finance App V2 token could not be decrypted"` — but the tenant is still imported with empty tokens and `ConnectionStatus = "NeedsReconnect"`. The root cause (wrong `ApplicationName`: `"FinanceApp.Api"` here vs possibly a different value in V2) is never diagnosed or surfaced. | `XeroLedgerServices.cs:60,94`; `Program.cs:19-21` — `ApplicationName` defaults to `"FinanceApp.Api"`. `XeroLedgerServices.cs:24` — protector purpose `"FinanceApp.Secrets.v1"`. | Closecore's import tooling validates the application-name and purpose string match before attempting decryption and surfaces a clear mismatch error. | On V2 import start, emit a log warning with the exact `DataProtection:ApplicationName` and purpose string so operators can cross-check against V2's config. Add a validation step that attempts to decrypt one token and fails fast with a clear error if the key-ring is incompatible, rather than importing broken records. |
| **Major** | One OAuth callback stores the **same** access token on both `XeroTenantConnection` and `XeroConnection` rows (legacy table) for every tenant returned. If a user has 3 tenants on one Xero org, the same token is written six times. The legacy `XeroConnection` table is never refreshed independently — it holds a stale copy after the next token rotation in `XeroTenantLedgerService.EnsureValidTokenAsync`. | `FinancialServices.cs:1684-1744` — writes both tables in the same `CompleteCallbackAsync` loop. `XeroLedgerServices.cs:609-634` — only updates `XeroTenantConnection`. `FinancialServices.cs:2887-2923` — only updates `XeroConnection`. | Best-in-class: single canonical token store; legacy tables are views or sync'd atomically. | Either remove `XeroConnection.EncryptedAccessToken`/`EncryptedRefreshToken` and always read from `XeroTenantConnection`, or after every token rotation in the ledger service, propagate the new token back to `XeroConnection` in the same `SaveChanges` call. |
| **Major** | `POST /api/xero/import-v2-tokens` has no idempotency guard for concurrent imports. The V2 import iterates tenants and calls `db.SaveChangesAsync` per iteration in a loop with no distributed lock. If two import requests run in parallel, both will `FirstOrDefaultAsync` on the same `TenantId` and then both issue `db.XeroTenantConnections.Add(...)` for the same entity, violating the unique index and causing a runtime exception. | `XeroLedgerServices.cs:83-120` — no lock around the upsert loop; `SaveChangesAsync` is deferred to `line 120` but `EF tracking state` is checked with `EntityState.Detached`. | Best-in-class: import endpoints are idempotent with optimistic concurrency or serialized via a queue. | Add a simple `SemaphoreSlim(1,1)` in `XeroTenantLedgerService` to serialize import calls, or move the upsert logic into a `UPSERT` SQL with `ON CONFLICT DO UPDATE` to make it inherently idempotent. |
| **Moderate** | No webhook/event-push for token revocation from Xero's side. If a Xero org admin removes the app in Xero's connected-apps UI, the tokens are silently invalid. The next sync attempt catches the failure via HTTP 401 — but because `IsTransient` excludes 401 (see Category 1), the error message is opaque and the tenant is not automatically marked `NeedsReconnect`. | `XeroBackfillServices.cs:116-121`; `XeroLedgerServices.cs:382-389`. | Fathom and Numeric subscribe to Xero's webhook for `AUTH_EVENT_TYPE = DISCONNECT` and immediately mark the connection `NeedsReconnect` with a clear UI message. | Register a Xero webhook endpoint (`POST /api/xero/webhook`), validate the `x-xero-signature` HMAC-SHA256 header, and on `DISCONNECT` events call `MarkNeedsReconnect`. Short-term: handle 401 as described in Category 1 to at least surface the failure cleanly. |
| **Minor** | Hardcoded absolute path in `appsettings.json` and code for the V2 DB. Both `appsettings.json:25` and `XeroLedgerServices.cs:782` embed `/Users/rickygarver/Projects/...` — this will fail on any other machine or CI environment. | `appsettings.json:25`; `XeroLedgerServices.cs:780-783`. | No peer product ships with dev absolute paths in config. | Move the fallback path to a relative path or remove it entirely; require `Xero:FinanceAppV2DbPath` to be set explicitly in environments where V2 import is needed. |

## Pillar verdict

- **Xero:** 🔴 — Two blockers (no revocation + concurrent refresh race) and four majors mean the Xero integration is unsafe for multi-user or multi-worker production use; token integrity is at risk under any concurrent load.

## Top 3 fixes (your categories only)

1. **Add a per-tenant refresh-lock.** Introduce a singleton `ConcurrentDictionary<string, SemaphoreSlim>` and guard all three `EnsureValidTokenAsync` implementations — this is the highest-probability data-corruption failure in production.
2. **Implement token revocation on disconnect.** Add `DELETE /api/xero/connections/{id}` that calls Xero's revocation endpoint and removes DB rows, satisfying Xero's integration requirements and eliminating ghost credentials.
3. **Remove the `"ey"` plaintext fallback in `Unprotect`.** A silent decryption bypass undermines the entire DataProtection layer; a `CryptographicException` must surface as `NeedsReconnect`, not return a cleartext credential.
